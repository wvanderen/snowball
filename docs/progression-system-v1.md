# Snowball Progression System v0.1

## 1. Purpose

This spec defines the first real progression layer for Snowball’s lattice game.

The goal is to make each sphere feel like it develops a unique mechanical identity through use, without tying mechanics to the real-life content of the sphere. A sphere named Music, Fitness, Study, Rest, or Home should not receive predefined gameplay behavior from that label. The user supplies meaning. The game supplies a flexible lattice playground.

Snowball’s core remains:

```txt
open app → tap sphere → session starts → real effort feeds the lattice
```

The game layer should deepen that loop without blocking it. The original product vision is clear that Snowball should prioritize quick starting, partial effort, consistency over intensity, and a non-punitive model of return after missed days.

---

# 2. Design thesis

> **Snowball does not classify your life. It gives your life a lattice where repeated attention becomes structure.**

Each sphere is:

```txt
a named life domain
a history of practice
a node in the lattice
a source of Energy
a holder of Momentum
a customizable progression object
```

The sphere’s real-life label does not determine its mechanics.

Instead:

```txt
practice in sphere → gain sphere XP → earn sphere points → invest in abstract paths → sphere gains behavior
```

So the role of a sphere emerges from use and upgrades.

A sphere can become:

```txt
a routing engine
a storage vessel
a milestone bloomer
a recovery anchor
a session ignition point
a network echo node
a memory archive
```

But those are build outcomes, not hardcoded categories.

# 3. Core progression loop

## Session loop

```txt
1. User taps a sphere.
2. Session starts immediately.
3. Time spent generates Sphere XP.
4. Active session generates Energy.
5. Momentum increases.
6. Daily milestone progress increases.
7. If milestone completes, sphere Blooms.
8. Energy flows through edges toward Center.
9. Glyphs, talents, and edge effects modify the flow.
```

## Upgrade loop

```txt
1. Sphere XP increases sphere level.
2. Sphere levels grant Sphere Points.
3. Sphere Points are spent in Sphere Paths.
4. Sphere Paths alter sphere, edge, glyph, and milestone behavior.
5. Energy is spent on lattice-wide/game upgrades.
6. Memory eventually preserves selected history through prestige.
```

## Long-tail loop

```txt
1. Spheres accumulate history.
2. History produces Memory through prestige or long-term milestones.
3. Memory unlocks permanent options.
4. Future lattice cycles begin with deeper structure.
```

# 4. Core resources

## XP

XP represents real focused effort.

```txt
1 focused minute = 1 Sphere XP
```

XP should remain grounded. It should not be heavily multiplied by late-game systems.

Used for:

```txt
sphere leveling
sphere talent points
ritual/domain history
prestige eligibility
```

## Energy

Energy is the main game currency and visual flow.

Generated by:

```txt
passive sphere production
active sessions
milestone Blooms
Charge releases
glyph triggers
routing effects
```

Spent on:

```txt
sphere upgrades
edge upgrades
new sphere slots
glyph slots
center upgrades
respecs
routing improvements
```

Energy may scale wildly over time. XP should not.

## Momentum

Momentum is the consistency health of a sphere.

```txt
range: 0–100
```

Generated by:

```txt
sessions
milestone completion
recovery effects
Anchor talents
glyph triggers
```

Lost through:

```txt
gradual decay when neglected
```

Momentum should bend, not shatter. It replaces punitive streak pressure as the main consistency signal.

## Charge

Charge is stored potential.

Generated by:

```txt
Charge path talents
glyphs
incoming Energy
active sessions
milestone preparation
```

Used for:

```txt
burst releases
milestone effects
edge activation
protective Anchor effects
future combo systems
```

## Memory

Memory is long-term history/prestige currency.

Generated by:

```txt
sphere history
milestones
high-level spheres
prestige
Archive path
comeback events
```

Used for:

```txt
permanent sphere upgrades
prestige inheritance
branch unlocks
remembered talents
momentum floors
respec flexibility
advanced glyph sockets
```

Memory is not required for MVP, but the system should leave room for it.

# 5. Sphere progression model

Each sphere has its own progression.

```ts
type Sphere = {
  id: string;
  name: string;
  position: LatticePosition;

  xp: number;
  level: number;
  availablePoints: number;
  spentPoints: number;

  momentum: number;
  charge: number;

  milestoneTargetMinutes: number;
  milestoneProgressMinutes: number;
  milestoneCompletedToday: boolean;

  pathAllocations: SpherePathAllocation[];
  glyphSockets: GlyphSocket[];
  connectedEdgeIds: string[];

  memory?: SphereMemory;
};
```

A sphere’s build is determined by its path allocations.

```ts
type SpherePath = "Flow" | "Charge" | "Bloom" | "Anchor" | "Spark" | "Echo" | "Archive";
```

For MVP, implement only:

```txt
Flow
Charge
Bloom
Anchor
```

Later add:

```txt
Spark
Echo
Archive
```

# 6. Sphere XP and leveling

Sphere XP is earned only by practicing in that sphere.

```txt
1 minute in sphere = 1 XP for that sphere
```

A session in Music does not level Fitness. A session in Home does not level Study.

This keeps the sphere’s growth tied to actual attention.

## Suggested level curve v0.1

| Level | Total XP Required | Approximate Meaning    |
| ----: | ----------------: | ---------------------- |
|     1 |                 0 | newly created          |
|     2 |                15 | first meaningful touch |
|     3 |                45 | a few sessions         |
|     4 |               100 | early pattern          |
|     5 |               180 | sphere is taking shape |
|     6 |               300 | established domain     |
|     7 |               475 | significant practice   |
|     8 |               725 | strong history         |
|     9 |              1050 | mature sphere          |
|    10 |              1500 | major domain           |

Each level grants:

```txt
+1 Sphere Point
```

Optional later: levels past 10 require prestige or Memory unlocks.

---

# 7. Sphere Points

Sphere Points are spent in Sphere Paths.

Rules:

```txt
Sphere Points are sphere-specific.
Points earned by Music can only be spent on Music.
Points earned by Fitness can only be spent on Fitness.
```

This is crucial. It means real time spent in a domain gives that domain mechanical agency.

## Starting point model

At level 1:

```txt
0 available points
```

At level 2:

```txt
1 available point
```

At level 10:

```txt
9 total earned points
```

This keeps early progression understandable.

# 8. Sphere Paths

Sphere Paths are abstract classlines.

They do not describe life content. They describe lattice behavior.

## Flow

Routing, movement, throughput.

Fantasy:

```txt
This sphere is good at moving power through the lattice.
```

## Charge

Storage, buildup, delayed release.

Fantasy:

```txt
This sphere accumulates potential and releases it at the right moment.
```

## Bloom

Milestone events, completion rewards, flowering.

Fantasy:

```txt
This sphere turns completion into visible power.
```

## Anchor

Stability, recovery, decay resistance.

Fantasy:

```txt
This sphere keeps itself and the lattice alive.
```

## Spark

Active-session ignition. Future branch.

Fantasy:

```txt
This sphere lights up when touched.
```

## Echo

Replication, spreading, network resonance. Future branch.

Fantasy:

```txt
This sphere makes effects ripple through the lattice.
```

## Archive

Memory, prestige, long-term inheritance. Future branch.

Fantasy:

```txt
This sphere remembers what it has been.
```

# 10. Emergent archetypes

Do not implement archetypes as fixed classes.

Instead, archetypes are descriptive labels inferred from path investment.

Examples:

```txt
Flow-heavy sphere = Conduit
Charge-heavy sphere = Vessel
Bloom-heavy sphere = Bloomer
Anchor-heavy sphere = Anchor
Flow + Bloom = Channeler
Charge + Bloom = Capacitor
Anchor + Bloom = Hearth
Flow + Charge = Reservoir
```

These labels can appear in the UI later, but they should not restrict mechanics.

Example UI:

```txt
Music
Level 6
Dominant Paths: Charge 3, Bloom 2, Anchor 1
Current Pattern: Capacitor
```

This is flavorful but not prescriptive.

# 11. Respec rules

Spheres should be flexible.

A sphere’s role in the user’s life can change, and the game should support that.

## MVP respec model

```txt
First respec per sphere is free.
Later respecs cost Energy.
Refund all spent Sphere Points.
```

Suggested cost:

```txt
Respec Cost = 25 Energy × spent points
```

Example:

```txt
Sphere has 6 spent points.
Respec cost = 150 Energy.
```

## Design language

Use gentle language:

```txt
Reconfigure Sphere
Refund Points
Reshape
Restore Previous Shape
```

Avoid punitive language:

```txt
Reset mistake
Penalty
Failure
```

## Future respec upgrades

Memory or Archive systems can unlock:

```txt
free weekly respec
saved builds
discounted refunds
remembered favorite build
partial point refund
```

# 12. Energy flow model

Each sphere generates Energy.

Energy travels through edges toward the Center.

For MVP, keep routing simple:

```txt
Energy generated by a sphere travels along shortest available path to Center.
If directly connected to Center, it flows directly.
If multiple equal paths exist, choose primary path or split evenly.
```

Later, routing can become strategic.

## Basic production formula

```txt
Sphere Energy = Base Production × Level Multiplier × Momentum Multiplier × Active Multiplier × Talent Modifiers × Glyph Modifiers
```

Suggested v0.1 values:

```txt
Base passive production: 0.01 Energy/sec
Active multiplier: 10x
Level multiplier: 1 + sphere level × 0.1
Momentum multiplier: 0.5 + momentum / 100
```

Example:

```txt
Level 4 sphere
Momentum 60
Passive production:

0.01 × 1.4 × 1.1 = 0.0154 Energy/sec
```

While active:

```txt
0.0154 × 10 = 0.154 Energy/sec
```

These numbers are placeholders. The important part is the formula shape.

# 13. Momentum model

Momentum ranges from:

```txt
0–100
```

## Session gain

Suggested:

```txt
+0.5 Momentum per focused minute
```

Cap:

```txt
cannot exceed 100
```

## Milestone gain

Suggested:

```txt
+10 Momentum on milestone completion
```

## Daily decay

Suggested:

```txt
Momentum decays by 10% of current value per neglected day
```

Anchor effects reduce this.

Example:

```txt
Momentum 70
No session today
Decay = 7
New Momentum = 63
```

With Anchor Rank 1:

```txt
Decay reduced by 5%
Decay = 6.65
New Momentum = 63.35
```

This is intentionally gentle.

## Momentum floor

Momentum floor prevents long-practiced or Anchor-invested spheres from collapsing to zero.

Base:

```txt
0
```

Anchor Rank 2:

```txt
+5 floor
```

Future Memory upgrades:

```txt
+5 to +25 floor depending on sphere history
```

---

# 14. Daily milestone and Bloom model

Each sphere has a daily milestone target.

Example:

```txt
20 minutes
```

Partial progress always counts.

Milestone completion triggers a Bloom once per day.

## Basic Bloom

Suggested formula:

```txt
Bloom Energy = milestone target minutes × active baseline × momentum multiplier × Bloom modifiers
```

Simpler MVP version:

```txt
Bloom Energy = milestone target minutes × 2
```

Then apply modifiers:

```txt
Bloom Rank 1: +10%
glyph effects
momentum effects
```

Example:

```txt
20 minute milestone
Base Bloom = 40 Energy
Bloom Rank 1 = +10%
Final Bloom = 44 Energy
```

Keep the first implementation simple. Tune later.

## Partial Bloom

With Bloom Rank 3:

```txt
first session of day grants Mini Bloom
Mini Bloom = 15% of expected full Bloom
```

This reinforces the core product promise: starting matters.

# 15. Charge model

Each sphere can store Charge.

Base max Charge:

```txt
100
```

Charge Path Rank 2:

```txt
Max Charge +50%
```

New max:

```txt
150
```

## Incoming Energy storage

With Charge Rank 1:

```txt
5% of incoming Energy becomes Charge
```

Important decision:

For MVP, stored Charge should not reduce the routed Energy. It should be copied, not diverted.

Why:

```txt
less punishing
easier to understand
more rewarding
```

Later, advanced talents can create tradeoffs.

## Charge release

With Charge Rank 3:

```txt
on milestone completion:
  release all stored Charge as Energy toward Center
  set Charge to 0
```

This creates the first satisfying combo:

```txt
store Energy → complete milestone → burst
```

# 16. Edge model v0.1

For MVP, edges are simple.

```ts
type Edge = {
  id: string;
  fromSphereId: string;
  toSphereId: string;
  level: number;
  throughputMultiplier: number;
  routingLoss: number;
};
```

Suggested defaults:

```txt
throughputMultiplier: 1.0
routingLoss: 0.0 for direct Center edges
routingLoss: 0.05 for non-Center edges
```

Flow talents can affect edges.

Later edge types:

```txt
Conduit
Gate
Lens
Valve
Bridge
Loop
Root
Thread
```

Do not implement yet unless the base routing feels too flat.

# 18. Modifier/effect system

Implementation should favor a generic effect model.

Avoid hardcoding all talent logic directly inside session or production functions.

Use declarative effects where possible.

```ts
type ModifierEffect =
  | {
      type: "MULTIPLY_OUTGOING_ENERGY";
      value: number;
    }
  | {
      type: "REDUCE_ROUTING_LOSS";
      value: number;
    }
  | {
      type: "ACTIVE_EDGE_THROUGHPUT_BONUS";
      value: number;
    }
  | {
      type: "STORE_INCOMING_ENERGY_AS_CHARGE";
      value: number;
    }
  | {
      type: "MULTIPLY_MAX_CHARGE";
      value: number;
    }
  | {
      type: "RELEASE_CHARGE_ON_MILESTONE";
      value: number;
    }
  | {
      type: "MULTIPLY_MILESTONE_BLOOM";
      value: number;
    }
  | {
      type: "BLOOM_NEIGHBOR_ENERGY_SHARE";
      value: number;
    }
  | {
      type: "FIRST_SESSION_MINI_BLOOM";
      value: number;
    }
  | {
      type: "REDUCE_MOMENTUM_DECAY";
      value: number;
    }
  | {
      type: "INCREASE_MOMENTUM_FLOOR";
      value: number;
    }
  | {
      type: "RETURN_AFTER_INACTIVITY_MOMENTUM_BONUS";
      value: number;
    };
```

Talents and glyphs both provide effects.

```ts
type TalentDefinition = {
  id: string;
  path: SpherePath;
  rank: number;
  name: string;
  description: string;
  effects: ModifierEffect[];
};
```

```ts
type GlyphDefinition = {
  id: string;
  name: string;
  description: string;
  socketType: "sphere" | "edge" | "center";
  effects: ModifierEffect[];
};
```

This gives us room to expand without rewriting the game loop every time.

# 20. UI principles

The progression UI should not compete with the start flow.

Home screen remains:

```txt
tap sphere → session starts
```

Progression is secondary.

Suggested UI structure:

```txt
Tap sphere once: start session.
Long press or detail button: open sphere detail.
Sphere detail: XP, Momentum, Charge, milestone, paths, glyph.
```

Do not force users into upgrade screens after every session.

Instead, use gentle indicators:

```txt
Music has 1 unspent point.
Music is ready to grow.
Choose how this sphere develops.
```

# 24. The buildable core

For the next implementation phase, the core system is:

```txt
Each sphere earns XP from sessions.
XP grants sphere-specific points.
Points are spent into abstract paths.
Paths modify Energy, Momentum, Charge, and Blooms.
The sphere’s role emerges from its build.
The user can respec.
Nothing is tied to the sphere’s real-life label.
```

That is the foundation.

It gives Snowball a real game spine without betraying the product vision.

The first build should focus on this:

```txt
Flow / Charge / Bloom / Anchor
3 ranks each
Sphere XP
Sphere Points
Charge
Milestone Bloom
Respec
Generic effect resolver
```

Once that feels good, the lattice can get much weirder.
